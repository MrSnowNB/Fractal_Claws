## KI-001 — `test_run_command_blocked` fails on Windows (bash not found)

**File:** `tests/test_tools.py`  
**Test:** `test_run_command_blocked_unix`  
**Platform:** Windows (win32)  
**First seen:** STEP-03 baseline gate  

### Root Cause
The original test used `bash -c "rm -rf /"` to verify the blocklist fires.
`bash` is not installed by default on Windows, so the subprocess call fails
before the blocklist even activates.

### Fix Applied
- Renamed to `test_run_command_blocked_unix` with `@pytest.mark.skipif(sys.platform == "win32")`
- Added `test_run_command_blocked_windows` using `cmd /c del /f /s /q c:\\`
- Added matching Windows patterns to `src/tools/terminal.py` blocklist

### Resolution Status
✅ Fixed — cross-platform coverage restored. Unix test skips cleanly on Windows.

## KI-002 — `output/audit.json` and `output/harness_result.json` missing on clean checkout

**File:** `tests/test_harness_artifacts.py`  
**Tests:** `test_runtime_artifact_exists[output/audit.json]`, `test_runtime_artifact_exists[output/harness_result.json]`  
**Platform:** All  
**First seen:** STEP-03 → pre-STEP-04 baseline gate  

### Root Cause
These files are **runtime outputs** generated by `agent/runner.py` executing
TASK-011 (the audit ticket). They are NOT committed to the repo. On a clean
checkout or after `git clean -fdx`, they do not exist.

The original `test_artifact_exists` parametrized test mixed these runtime
artifacts with static committed artifacts (`output/word_freq.py`,
`output/report.md`), causing 2 failures on every clean run.

### Fix Applied
- Split `ARTIFACTS` into `ARTIFACTS_STATIC` (always enforced) and
  `ARTIFACTS_RUNTIME` (skipped unless `HARNESS_RUN=1` env var is set)
- Runtime artifact tests use `pytest.skip()` when file absent and
  `HARNESS_RUN != 1`
- CI / full harness validation: set `HARNESS_RUN=1` after running
  `python agent/runner.py --no-prewarm`

### Resolution Status
✅ Fixed — default `pytest tests/ -v` run is now clean.
  Full enforcement: `HARNESS_RUN=1 pytest tests/test_harness_artifacts.py`

## KI-003 — `echo` is a shell built-in on Windows, not an executable

**File:** `tests/test_tools.py`  
**Tests:** `test_run_command_basic`, `test_run_command_elapsed_ms`  
**Platform:** Windows (win32)  
**First seen:** pre-STEP-04 baseline gate  

### Root Cause
`echo` on Windows is a **cmd.exe built-in** — there is no `echo.exe` on the
PATH. `subprocess.run(["echo", "hello"])` without `shell=True` raises
`FileNotFoundError: [WinError 2] The system cannot find the file specified`
because Python tries to find an `echo` executable and fails.

The same applies to any bare shell built-in called as a list argument to
`subprocess.run` on Windows: `echo`, `dir`, `type`, `set`, `cls`, etc.

### Fix Applied
- Added `_echo_cmd(*words)` helper in `test_tools.py` that returns
  `["cmd", "/c", "echo"] + words` on win32 and `["echo"] + words` on Unix
- `test_run_command_basic` and `test_run_command_elapsed_ms` now use `_echo_cmd()`
- No changes to `terminal.py` or `run_command()` — the fix is in the test layer
  since this is a test authoring issue, not a runtime issue

### Resolution Status
✅ Fixed — both tests pass on Windows and Unix.

## KI-004 — `test_audit_json_schema` / `test_harness_result_passed` — stale harness output

**File:** `tests/test_harness_artifacts.py`  
**Tests:** `test_audit_json_schema`, `test_harness_result_passed`  
**Platform:** All  
**First seen:** pre-STEP-04 baseline gate  

### Root Cause
`output/audit.json` exists on disk but contains a partial/stale run result:
- `report_char_count` is `None` (TASK-011 audit script ran but didn't capture char count)
- `harness_result.json` shows `failed_checks: ['check1', 'check3']` (audit found failures)

This means the harness runner executed TASK-011 but the audit script itself
had logic errors or incomplete data at the time it ran. The JSON files exist
but contain bad data from an incomplete prior run.

### Fix Applied
No code fix applied here — this requires re-running the harness after the
underlying audit ticket (TASK-011) is corrected. These tests are already
gated behind `HARNESS_RUN=1` (KI-002 fix), so they skip in default runs.

### Resolution Path
1. Fix TASK-011 audit script so it correctly captures `report_char_count`
2. Run `python agent/runner.py --no-prewarm` to regenerate artifacts
3. Verify with `HARNESS_RUN=1 pytest tests/test_harness_artifacts.py`

### Resolution Status
⏳ Pending — requires TASK-011 audit script fix + full harness re-run.
  Blocked by: TASK-011 not yet corrected.
  Tracked as: pre-STEP-04 known issue. Will be addressed in STEP-04 or dedicated fix ticket.

## Gate Protocol Reference

```
Default gate (every step, every commit):
  pytest tests/ -v
  Expected: 0 failures
  Expected skips: 1 (platform-specific blocked test)

Harness gate (after runner.py execution only):
  python agent/runner.py --no-prewarm
  HARNESS_RUN=1 pytest tests/test_harness_artifacts.py -v
  Expected: 0 failures

Platform notes:
  Windows:
    - test_run_command_blocked_unix  → SKIP (expected, KI-001)
    - test_run_command_blocked_windows → RUN
    - echo commands use cmd /c echo (KI-003)
  Unix/macOS:
    - test_run_command_blocked_unix  → RUN
    - test_run_command_blocked_windows → SKIP (expected)
    - echo commands use bare echo
```
